# spec-delivery-lite 方案设计

## 1. 项目定位

`spec-delivery-lite` 是一个**全新的、独立的 AI 编程工作流项目**，不是 OpenSpec 的 fork、插件、schema 或目录约定的改造版。

它融合两个来源的互补能力：

- 从 **OpenSpec** 吸收：以变更为中心的 artifact、可追溯的生命周期、规格与实现之间的关联、变更完成后的归档。
- 从 **superpowers** 吸收：风险驱动的设计澄清、证据链调试、审慎处理评审、完成前验证、依赖感知的并行协作。

项目目标不是复刻任一上游项目，而是在单仓库或中小型项目中提供更少命令、更少文件、但覆盖“想法 → 规格 → 实现 → 验证 → 归档”完整闭环的 Skill 与 CLI。

```text
OpenSpec 的变更可追溯性
            +
superpowers 的执行质量门禁
            ↓
spec-delivery-lite：轻量、完整、风险自适应的变更交付系统
```

## 2. 问题与目标

### 2.1 要解决的问题

直接让 AI 编码容易出现四类问题：需求只存在于对话中、实现偏离原意、遇错后猜测式叠补丁、没有新鲜证据就声称完成。

完整 OpenSpec 和 superpowers 能分别缓解这些问题，但同时采用会带来重复 artifact、过多 workflow、强制 worktree/子代理编排和不必要的文档负担。

### 2.2 产品目标

1. 为每个值得记录的变更保存范围、验收条件、任务和决策。
2. 根据风险自动提升或降低设计深度，而不是强制固定四阶段。
3. 把实现、调试、评审、并行和验证纳入同一个变更上下文。
4. 以测试、构建和验收条件追踪作为完成结论的依据。
5. 不依赖 OpenSpec CLI、OpenSpec 目录、Store 或任何上游运行时。

### 2.3 非目标

- 不兼容或执行 OpenSpec 的 schema、CLI 命令和 Store 协议。
- 不复刻 superpowers 的全部 Skill、强制 worktree 或逐任务双阶段子代理审查。
- 不替代 CI/CD、部署、项目管理系统或 Git 托管平台。
- 不要求每个小修复都创建变更文档。

## 3. 设计原则

```text
风险决定设计深度；
规格决定实现边界；
依赖决定是否并行；
证据决定结论强度。
```

- **独立命名空间**：所有状态均位于 `.spec-delivery-lite/`，不写入 `openspec/`。
- **一个变更，一份事实来源**：对话是输入，artifact 是可恢复的持久上下文。
- **最少但足够的文件**：默认两个 artifact；高风险时增加明确目的的 artifact。
- **质量策略内建**：五项 superpowers 能力贯穿生命周期，不作为可有可无的附录。
- **人工拥有边界决策**：范围、不可逆操作、架构取舍和待决策评审意见由用户确认。

## 4. 概念模型与文件结构

### 4.1 目录结构

```text
.spec-delivery-lite/
├── config.yaml                 # 项目级规则、测试/构建命令、目录约定
├── changes/
│   └── <change-name>/
│       ├── change.yaml          # 格式版本、风险、状态、确认与验收项状态
│       ├── brief.md            # 目标、范围、非目标、验收条件
│       ├── design.md           # 按需创建：高风险技术决策
│       ├── tasks.md            # 实施任务和进度
│       └── verification.md     # 验证证据与规格追踪
└── archive/
    └── YYYY-MM-DD-<change-name>/
```

### 4.2 Artifact 的渐进式使用

| Artifact | 默认 | 用途 | 创建条件 |
|---|---:|---|---|
| `change.yaml` | 是 | 格式版本、风险、生命周期、确认与验收项状态 | 任何需要记录的变更 |
| `brief.md` | 是 | 目标、范围、非目标、验收条件 | 任何需要记录的变更 |
| `tasks.md` | 是 | 可勾选的垂直切片、关联验收条件、验证命令 | 任何需要记录的变更 |
| `design.md` | 否 | 方案比较、架构/API/数据决策、风险与回滚 | 存在重要技术取舍或高风险变更 |
| `verification.md` | 完成时 | 验收条件到证据的追踪、未解决风险 | 准备宣称完成或归档 |

`change.yaml` 是生命周期的唯一权威记录；`brief.md` 是范围与验收条件的唯一权威位置；任务和验证只能通过稳定的 `AC-##` 标识引用验收条件。

这不是简单地把 OpenSpec 的 `proposal/specs/design/tasks` 改名：

- `brief.md` 同时承担变更提案和可验收需求的角色；
- `design.md` 是条件化的，不为小改动强制创建；
- `verification.md` 把 OpenSpec 的实现-规格核对与 superpowers 的证据规则持久化；
- `tasks.md` 是唯一实施进度来源，不另建 superpowers plan。

### 4.3 关键模板

`brief.md`：

```markdown
# Change: <change-name>

## 目标与范围
要解决的问题、目标用户和本次边界。

## 非目标
明确本次不会做什么。

## 验收条件
- AC-01：给定 <前置条件>，当 <用户动作> 时，系统应 <可观察结果>。
  - 验证类型：automated / manual。
- 失败行为、兼容性或性能约束（如适用）。

## 影响面
受影响的模块、公开接口、数据或外部服务。
```

`tasks.md`：

```markdown
# Tasks

- [ ] 状态与持久化
  - 验收：刷新后保留选择。
  - 验证：`npm test -- theme`。
- [ ] 界面切换入口
  - 验收：用户切换后页面更新。
  - 验证：`npm test -- theme`。
```

`verification.md`：

```markdown
# Verification: <change-name>

| 验收条件 | 证据 | 结果 |
|---|---|---|
| 刷新后保留选择 | `npm test -- theme`，测试名称：… | 通过 |
| 用户可切换主题 | 浏览器场景验证 / e2e 测试：… | 通过 |

## 完整检查
- 测试：命令、退出码、失败数。
- 构建/静态检查：命令、退出码、警告。
- 已知限制或未验证项。
```

`change.yaml`：

```yaml
format_version: 1
status: planned # planned | approved | implementing | blocked | verified
risk: tracked # direct | tracked | high
approval:
  required: false
  confirmed_by: null
acceptance:
  - id: AC-01
    status: pending # pending | verified | waived
```

## 5. 生命周期

```text
发现请求
  │
  ├─ 低风险且明确 ──→ 直接实施 → 验证 → 汇报
  │
  └─ 需要变更记录
       │
       ├─ 评估是否需要深入 brainstorming
       ├─ 建立 / 更新 brief.md
       ├─ 必要时建立 design.md 并等待确认
       ├─ 创建 tasks.md
       ├─ 实施与调试
       ├─ 评审和（条件化）并行协作
       ├─ 重新验证并写入 verification.md
       └─ 归档
```

生命周期是可回退的：实施发现规格或设计错误时，先更新 artifact，再让任务和代码追上新的已确认事实；不会强迫用户按不可逆阶段推进。

Tracked change 的状态仅可按以下规则转换：`planned → approved / implementing / blocked`，`approved → implementing / blocked`，`implementing → blocked / verified`，`blocked → 原工作状态`。只有所有验收项都已有新鲜证据、或已记录用户接受的豁免时，才可进入 `verified` 并归档。

## 6. 融合的质量能力

以下五项来自 superpowers 的核心思想，但以轻量规则融入所有阶段；它们不是项目的全部功能，而是本项目的质量层。

### 6.1 风险驱动 brainstorming

仅在以下任一情况成立时进行完整澄清、方案比较与确认：

1. 需求、范围或成功标准不清楚；
2. 存在多条差异明显的实现路径；
3. 会改变架构、公开行为或数据契约；
4. 决策代价高、难以回滚，或涉及安全、权限、隐私、数据迁移、不可逆外部操作。

以下是强制 High-risk 的可检查边界：数据迁移或删除；安全、权限或隐私；公开 API、配置格式或跨模块契约且存在兼容性取舍；生产/外部写入；不可逆或高成本决策。其他单模块、可逆且不改变用户可见行为或契约的窄改动可为 Direct；bug 修复和用户可见行为至少为 Tracked。

完整模式应提出 2--3 个方案、给出取舍与推荐，并把确认后的决策写入 `brief.md` 或 `design.md`。小而明确、已批准且可安全回滚的任务直接继续。

### 6.2 基于证据的调试

```text
复现问题 → 定位出错边界 → 提出一个假设 → 最小实验 → 修复源头 → 回归验证
```

- 最小实验必须能区分假设成立与否。
- 用户只要求分析原因时，止于诊断结果，不修改代码。
- 同一方向连续两次假设失败时，回到复现边界、日志和架构假设，不继续叠加补丁。

### 6.3 审慎处理代码评审

| 分类 | 处理 |
|---|---|
| 接受 | 意见正确，直接处理并验证 |
| 调整 | 问题存在，但采用更适合当前代码的修复方式 |
| 反驳 | 意见与代码事实、兼容性或已确认需求冲突，给出证据 |
| 待决策 | 涉及产品、架构或范围，交由用户决定 |

安全、数据损坏和关键正确性问题不得仅以“反驳”关闭，必须提供充分替代证据或升级为待决策。

### 6.4 完成前重新认证

在宣称“完成、修复、测试通过、可合并”等结论前：明确结论 → 找到检查 → 在最终修改后执行 → 阅读退出码、失败数和警告 → 只报告证据支持的范围。

对可自动化验证的行为变更，新增或更新测试；对 bug 修复，优先先让测试复现原问题。最终将所有验收条件映射到测试、构建输出、代码路径或手动场景验证，写入 `verification.md`。

### 6.5 有条件的并行

只有在以下条件都成立时并行：按独立问题域拆分、没有先后依赖、修改范围和共享资源不重叠、同轮发出、返回后检查真实产物与冲突、分别验证后再组合验证。

主代理负责范围、边界和集成决策；子代理负责资料检索、独立调查或互不重叠的实现工作。

## 7. Skill 的渐进式披露设计

项目对外只提供一个入口 Skill：`spec-delivery-lite`。这不代表所有流程细节都写进一个 `SKILL.md`；入口文件应作为生命周期路由器，按当前事件加载对应的参考模块。

采用单入口而非多个独立 Skill 的原因是：同一个变更会在实施中切换到调试、评审、验证或并行状态。由同一个入口持有 change 上下文，能避免不同 Skill 之间丢失范围、验收条件和任务进度。

```text
spec-delivery-lite/
├── SKILL.md
├── agents/
│   └── openai.yaml
├── references/
│   ├── artifacts.md
│   ├── brainstorming.md
│   ├── debugging.md
│   ├── code-review.md
│   ├── verification.md
│   └── parallel-work.md
└── assets/
    └── templates/
        ├── brief.md
        ├── design.md
        ├── tasks.md
        └── verification.md
```

### 7.1 各层加载责任

| 层级 | 内容 | 加载时机 |
|---|---|---|
| 元数据 | Skill 名称与触发描述 | 始终可见，用于识别软件变更请求 |
| `SKILL.md` | 风险判定、当前阶段识别、文件约定、引用路由、全局不可突破规则 | Skill 触发时 |
| `references/*.md` | 特定场景的详细操作步骤与边界 | 仅在对应场景发生时 |
| `assets/templates/*.md` | 要复制到目标项目的 artifact 模板 | 创建或更新对应 artifact 时 |

`SKILL.md` 应控制在约 100--150 行，只保留：

1. `.spec-delivery-lite/` 的基础约定；
2. 低、中、高风险判定；
3. 根据事件加载 reference 的路由；
4. 全局规则，例如高风险决策须确认、最终修改后必须重新验证。

### 7.2 引用路由

| 当前情形 | 读取模块 | 行为 |
|---|---|---|
| 低风险、明确、可逆的小改动 | 无额外模块；完成前读取 `verification.md` | 直接实施与验证，不创建 change |
| 需要创建或更新 change | `artifacts.md` | 创建/更新 brief、按需 design、tasks 或 verification artifact |
| 需求不清、多方案、架构/契约变化或高回滚成本 | `brainstorming.md` | 澄清、比较方案、请求关键决策确认 |
| 出现报错、测试失败或行为与预期不符 | `debugging.md` | 按证据链复现、假设、实验、修复和回归 |
| 收到代码评审意见 | `code-review.md` | 分类为接受、调整、反驳或待决策 |
| 准备宣称完成、提交或归档 | `verification.md` | 重新执行检查并建立验收条件到证据的追踪 |
| 考虑启动多个 Agent | `parallel-work.md` | 检查独立性、资源隔离与组合验证要求 |

这使简单任务只加载入口和完成验证规则；复杂任务才会随着状态变化，逐步加载 artifact、brainstorming、调试、并行和验证细节。

## 8. 产品形态

### 8.1 Skill

核心 Skill 名为 `spec-delivery-lite`，负责在对话中识别当前生命周期阶段、读取/更新本项目 artifact，并执行质量门禁。

```yaml
---
name: spec-delivery-lite
description: Use when planning, implementing, debugging, reviewing, validating, or safely parallelizing a software change. Manage the change lifecycle in .spec-delivery-lite, keep scope and acceptance criteria explicit, and apply risk-driven design and evidence-based verification.
---
```

### 8.2 CLI（后续版本）

CLI 只提供本项目状态和 artifact 操作，不模仿 OpenSpec 的完整命令面：

```text
spec-delivery-lite init
spec-delivery-lite new <change-name>
spec-delivery-lite status [change-name]
spec-delivery-lite verify <change-name>
spec-delivery-lite archive <change-name>
```

Skill 可以在没有 CLI 的情况下按文件约定工作；CLI 用于初始化、结构校验、状态汇总和归档的确定性操作。

## 9. 对 OpenSpec 的取舍评估

OpenSpec 的设计目标是支持多种 AI 工具、团队、仓库和可定制工作流。以下能力在该目标下是合理的；但对本项目的单仓库、轻量定位，其中一部分会成为额外复杂度，而非直接价值。

| OpenSpec 部分 | 评估 | spec-delivery-lite 的处理 | 理由 |
|---|---|---|---|
| 以变更为中心组织工作 | 保留 | 独立的 change 目录与归档 | 将对话意图变为可审阅、可恢复的持久事实，是核心价值 |
| proposal / specs / design / tasks 四 artifact | 小变更偏重 | 默认 `brief.md + tasks.md`，高风险时增加 `design.md` | proposal 与 specs 在小变更中常重叠；design 只在有实质技术取舍时才有价值 |
| `explore / propose / apply / archive` 核心闭环 | 保留概念，不复制命令 | 一个 Skill 根据生命周期选择模式 | 探索、记录、实施、收尾是必要能力，但不需要用户记忆多个 workflow 名 |
| 扩展 workflow（new、continue、ff、update、verify、sync 等） | 不纳入 v1 | 由 Skill 和少量 CLI 子命令承担 | 细粒度 artifact 控制与教学流程会提高认知负担，普通变更不需要 |
| 动态 schema、artifact 依赖与 instructions/status 协议 | 不纳入 v1 | 固定、透明的文件结构和模板 | 它们服务可编程工作流与多工具兼容；轻量项目应让文件本身即可理解 |
| Store 与跨仓库协作 | 不纳入 v1 | 限定本地项目状态 | 跨团队共享规格有价值，但不应成为单仓库工作流的运行时依赖 |
| delta specs 与同步长期主规格 | 不纳入 v1 | 归档变更记录，不维护第二份主规格 | 小项目中同步会带来额外维护；未来可作为独立扩展能力评估 |
| 规格与实现核对 | 保留 | `verification.md` 追踪验收条件与证据 | 测试全绿不等于需求完成；该能力与证据验证互补 |

因此，本项目借鉴 OpenSpec 的**信息组织模型**，不复制其面向通用性的**工作流引擎**。

## 10. 明确不采用的上游能力

| 来源 | 不采用 | 原因 |
|---|---|---|
| OpenSpec | Store、跨仓库协作协议、自定义 schema、动态 artifact 依赖、delta spec 同步、十余独立 workflow、OpenSpec CLI 兼容层 | 超出轻量单项目场景，增加学习、配置和维护成本 |
| superpowers | 强制 worktree、强制逐任务提交、逐任务双阶段子代理评审、额外 specs/plans 文档 | 与本项目 artifact 重复，且对简单变更摩擦过高 |

## 11. 实施路线

1. 创建可安装的入口 Skill：`SKILL.md` 和 `agents/openai.yaml`。
2. 编写六个按需 reference 模块，以及 `brief.md`、`design.md`、`tasks.md`、`verification.md` 模板。
3. 用三类真实场景前向验证：低风险小修复、普通功能、跨模块/高风险变更；检查每类场景只加载必要模块。
4. 实现最小 CLI：`init`、`new`、`status`、`verify`、`archive`。
5. 根据验证结果收紧触发条件、模板字段和验证规则，而非增加新的流程步骤。

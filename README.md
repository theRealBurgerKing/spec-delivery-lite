# spec-delivery-lite

本项目是一个面向中小型项目的轻量软件变更交付 Skill。它将变更范围、验收条件、实施任务和验证证据保存在项目内，帮助 AI 在“想法 → 设计 → 实施 → 验证 → 归档”之间保持可追溯的工作上下文。本项目借鉴了Openspec和Superpowers的部分思想。

## 适用场景

- 为功能开发、bug 修复或跨模块改动保留可恢复的工作上下文。
- 在架构、数据、权限、公开接口等高风险改动前完成方案比较与用户确认。
- 在宣称完成前，将每项验收条件映射到新鲜的测试、构建输出或手动验证证据。
- 在多个 Agent 并行工作前检查任务独立性、资源冲突和组合验证边界。

对单文件、可回滚且不改变用户可见行为或公开契约的小改动，Skill 不会强制创建变更文档。

## 核心原则

```text
风险决定设计深度；
规格决定实施边界；
依赖决定是否并行；
证据决定结论强度。
```

## 变更目录

需要追踪的变更保存在目标项目的 `.spec-delivery-lite/`：

```text
.spec-delivery-lite/
├── changes/
│   └── <change-name>/
│       ├── change.yaml        # 风险、生命周期、审批和验收项状态
│       ├── brief.md           # 目标、范围、非目标、验收条件与影响面
│       ├── design.md          # 按需创建：重要技术决策及回滚影响
│       ├── tasks.md           # 可验证的实施任务
│       └── verification.md    # 验收条件到证据的映射
└── archive/
    └── YYYY-MM-DD-<change-name>/
```

`change.yaml` 是生命周期的权威记录。每项验收条件使用稳定的 `AC-##` 标识；任务和验证记录只能引用该标识，避免范围、任务和证据相互脱节。

## 风险分级

| 等级 | 适用情况 | 要求 |
|---|---|---|
| Direct | 范围窄、可回滚，且不影响用户可见行为、公开契约、数据、安全、权限或架构 | 直接实施，但完成前仍须验证 |
| Tracked | 多步骤工作、bug 修复、用户可见行为变更或需要保留的行为 | 创建并维护变更 artifact |
| High-risk | 数据迁移/删除、安全、权限、隐私、公开契约兼容性取舍、外部写入、不可逆或高成本决策 | 比较方案、记录决策并取得用户确认 |

Tracked 变更使用 `planned`、`approved`、`implementing`、`blocked`、`verified` 状态。只有所有验收项都有新鲜证据，或已记录用户明确接受的豁免，才可进入 `verified` 并归档。

## 安装与使用

将本目录放入 Codex 的 Skills 目录，或在 Codex 中以本地 Skill 方式加载。之后可以显式调用：

```text
使用 $spec-delivery-lite 实现“记住用户主题选择”，并在完成前验证刷新后仍然生效。
```

也可以直接提出软件变更请求；Skill 会根据风险和当前变更状态选择合适流程。

## 内容结构

- [SKILL.md](SKILL.md)：唯一入口，负责分类请求、路由 reference 和执行全局约束。
- `references/`：按需读取的变更 artifact、方案澄清、调试、评审、验证和并行协作规则。
- `assets/templates/`：创建变更时复制的中文模板。
- [方案设计.md](方案设计.md)：设计动机、取舍和实施路线。

## 质量约束

- 高风险决策实施前必须有用户确认。
- bug 修复优先用测试复现；调试使用“复现 → 假设 → 最小实验 → 根因修复 → 回归验证”。
- 评审意见先根据代码、测试和已批准范围分类，再决定接受、调整、反驳或交由用户决策。
- 最终修改后重新执行相关检查；不能运行的检查只能作为未验证缺口或经用户接受的豁免报告。
- 并行任务不得共享文件、接口、测试夹具或外部资源，且必须进行组合验证。
